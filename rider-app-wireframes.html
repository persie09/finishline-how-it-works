1. Login Screen

Screen: FL-74

Purpose

Single entry point. Phone number only — no email, no password. Riders don't remember passwords.

Workflow

1
Rider opens app
2
Enters 10-digit mobile number
3
Taps "Get OTP" → API sends SMS
4
Redirects to OTP screen

Design Notes

  • Country code locked to +91 (India-only V1)
  • Phone field auto-formats with space: 98765 43210
  • CTA is full-width, high contrast green — impossible to miss
  • Logo + "Rider Partner" sets context (not customer app)
  • T&C link required legally, small and non-blocking

Error States to Design

  • Invalid number (< 10 digits): inline error below field
  • Rate limited (too many OTPs): toast message with cooldown
  • Network failure: retry prompt
  • Number not registered + new user: proceed to onboarding
  • Number registered + existing user: proceed to shift home
Keyboard should auto-open on screen load. Number pad only, not full keyboard.

2. OTP Verification

Screen: FL-74

Workflow

1
6-digit OTP auto-reads from SMS (Android)
2
If auto-read fails, rider types manually
3
Auto-submits when 6 digits entered
4
New user → Profile Setup (onboarding)
5
Existing user → Shift Home
6
Existing + unsettled COD → COD Settle screen (blocked)

Design Notes

  • 6 individual boxes, not one input field — clearer for low-literacy users
  • Auto-advance cursor to next box
  • Resend timer: 30 seconds, then "Resend" becomes tappable
  • "Back" changes phone number
  • Phone number shown in bold — rider confirms it's correct
Critical gate: if rider has unsettled COD from last shift, login succeeds but immediately shows COD settle screen. Cannot skip.

3. Profile Setup

Screen: FL-75 · Step 1/4

Workflow

1
First-time user lands here after OTP
2
Fills name, photo, vehicle type, bank details
3
Taps "Next" → Documents screen

Design Notes

  • Progress stepper at top — rider knows 4 steps total
  • Photo: camera or gallery picker. Circular crop.
  • Vehicle type: big tap targets, radio-style selection
  • Bank details: needed for payouts. Validate IFSC format inline.
  • All fields required. CTA stays disabled until all filled.
  • Data persists if rider leaves and returns — save draft on every field blur
Name must match Aadhaar. Show hint: "As on your Aadhaar card"

4. Document Upload

Screen: FL-76 · Step 2/4

Workflow

1
Aadhaar: DigiLocker (preferred) or camera upload
2
PAN: camera capture (front side)
3
DL: camera capture (front + back)
4
Selfie: front camera, liveness check placeholder

Design Notes

  • Each doc shows status: Verified (green), Pending (amber), Not uploaded (grey)
  • DigiLocker for Aadhaar = one-tap verification, shows masked number
  • Camera: overlay guide frame so rider captures correctly
  • Upload progress indicator per document
  • Retry on blurry/unclear photos
  • Can proceed with PAN pending — gets verified async
DigiLocker integration is the gold standard. If rider doesn't have DigiLocker, fallback to manual photo + async verification (24-48h).

5. Contractor Agreement

Screen: FL-76 · Step 3/4

Workflow

1
Rider scrolls through agreement
2
Checks "I agree" checkbox
3
Taps "Sign via DocuSign" — webview opens
4
Signs on DocuSign → callback → Step 4

Design Notes

  • Agreement is scrollable, not a PDF download
  • Checkbox + CTA are both required (legal compliance)
  • DocuSign opens in-app webview, not external browser
  • On DocuSign callback: auto-advance to next step
  • Keep it short visually — riders won't read 10 pages

6. Pending Approval

Screen: FL-76

Workflow

1
All 4 steps complete → application submitted
2
Background verification runs async (24-48h)
3
Store manager reviews + approves
4
Push notification: "You're approved!"
5
Rider opens app → lands on Shift Home

Design Notes

  • This is a dead-end screen — rider can only wait
  • Show checklist of what's done + what's pending
  • Estimated time: "Usually 24-48 hours"
  • No logout button needed — session persists
  • Pull-to-refresh to check status
  • Push notification on approval (high priority)
If BGV fails, show "Verification issue — contact support" with WhatsApp link. Don't show rejection details.

7. Shift Home (Main Screen)

Screen: FL-77

Purpose

The rider's home base. Shows shift status, hours progress, quick stats, and the primary CTA: "Scan QR to get orders."

Workflow

1
Rider opens app / returns from delivery
2
Sees: shift window, mandatory blocks, hours bar
3
Primary action: Scan QR to mark "ready for orders"
4
Secondary info: deliveries, earnings, cash in hand

Design Notes

  • Hours progress bar — the most important element. Rider knows exactly where they stand.
  • Mandatory blocks — red dots + Done/Active/Later status. Rider sees at a glance if they've met requirements.
  • Cash in hand — amber when approaching threshold, red when blocked
  • Tip card — soft nudge for distribution constraints. Not a penalty warning, a helpful tip.
  • QR button — BIG, yellow, unmissable. This is THE action.
  • Bottom nav: 3 tabs only. Shift (home), Earnings, Profile.
If rider hasn't checked in yet, the shift card shows "Not Checked In" state with QR CTA even more prominent. No stats shown until checked in.

8. QR Scan

Screen: FL-78

Purpose

QR scan = "I'm physically at the store and ready." This is the check-in mechanism AND the order-readiness signal.

Workflow

1
Camera opens with QR frame overlay
2
Rider points at store QR code
3
Scan success → haptic feedback + chime
4
First scan of day: also triggers selfie + uniform check
5
→ Ready State (waiting for assignment)

Design Notes

  • Dark background — camera dominates
  • Animated scan line gives feedback that scanner is active
  • Corner brackets show where to aim
  • Torch toggle for dark stores
  • Gallery option: fallback if camera has issues
First scan of the day triggers: (1) selfie capture, (2) bags/t-shirt verification (photo). These happen AFTER successful QR scan, before entering Ready state.

9. Ready State

Screen: FL-78

Purpose

Rider has scanned QR. They're in the assignment queue. Now they wait.

Workflow

1
Rider waits at store. Screen shows "Waiting for Order"
2
Assignment engine matches order → push notification
3
→ Order Assigned screen (auto-navigates)
4
Alternatively: rider can "Take a Break" (leaves queue)

Design Notes

  • Big green checkmark with pulse animation — "you're good"
  • Queue position — rider knows they're #2 in line (FIFO transparency)
  • Idle time shown — used for tiebreak in assignment
  • "Take a Break" button is secondary (outline) — not the primary action
  • No way to leave this screen except: get assigned, take break, or check out
Keep-alive: app pings server every 30s. If rider's phone dies or they leave app for >5 min, auto-mark as "away" and remove from queue.

10. Order Assigned

Screen: FL-79

Purpose

Order has been assigned. Rider sees what to pick up, where to deliver, how much they'll earn. One action: "Pick Up & Navigate."

Workflow

1
Push notification + screen auto-opens
2
Rider sees: items, distance, ETA, payout, COD status
3
Rider picks items from store shelf / counter
4
Taps "Pick Up & Navigate" → Active Delivery

Design Notes

  • Green header — visual urgency, this is an active assignment
  • 3 key numbers up top: distance, ETA, payout. Rider's decision info.
  • Items list with checkboxes — rider verifies they have everything
  • COD badge is prominent (amber). Rider must know to collect cash.
  • Customer address visible but not full detail yet
  • No "reject" button — assignment is locked. (V1 decision)
V1: No reject/decline. Assignment is final. If there's an issue, rider contacts store manager. V2 may add decline with penalty.

11. Active Delivery

Screen: FL-79

Purpose

Rider is en route. Map + key info + actions (arrived, call, navigate).

Workflow

1
Map shows route to customer
2
Rider follows navigation (Google Maps deep link)
3
Arrives → taps "Arrived"
4
→ Proof of Delivery screen

Design Notes

  • Map takes top 50% — rider needs visual route
  • Distance + ETA overlay on map (bottom-left)
  • 3 bottom buttons: Arrived (primary), Call, Navigate
  • Call: opens phone dialer with customer number
  • Navigate: deep links to Google Maps with destination
  • Order card below map: order ID, customer, items, COD status
  • Must work one-handed while riding (big touch targets)
GPS tracking: app sends location every 15s while on delivery. Used for ETA updates to customer and for distance calculation.

12. Proof of Delivery

Screen: FL-79

Purpose

Rider confirms delivery + collects COD. Two things happen on this screen: payment collection and delivery proof.

Workflow

1
If COD: collect cash amount, tap "Cash" or "Paid Online"
2
Delivery proof: photo OR customer OTP (one required)
3
Tap "Complete Delivery"
4
→ Back to Ready state (if still on shift) or Shift Home

Design Notes

  • COD section is top and prominent — rider must collect first
  • "Cash" vs "Paid Online" toggle — clear choice
  • Delivery proof: photo of package at door OR 4-digit OTP from customer
  • OTP shows as "verified" state (green) when customer shares it
  • "Complete Delivery" is final — cannot undo
If COD amount is wrong (customer disputes), rider should contact manager. V2: add "Amount dispute" flow.

13. Batch Assigned

Screen: FL-80

Purpose

Rider gets a batch of 2-3 orders. Needs to see: total route, per-stop details, combined payout.

Workflow

1
Assignment: "2 Orders — 3 Segments" (batch notification)
2
Rider sees route with stops in sequence
3
Each stop: customer, address, items, COD status
4
Taps "Start Route" → navigates to Stop 1

Design Notes

  • Green header with "BATCH ORDER" label — visually distinct from single order
  • Segments count shown (relates to pay: 2 orders = 3 segments)
  • Stop cards with left border: current (amber), upcoming (green), done (grey)
  • Each stop shows: customer name, COD/Prepaid badge, distance, items
  • Combined stats: total distance, total ETA, total payout
Batch sequence is optimized by the system (shortest path). Rider follows the suggested order. V2: allow rider to reorder stops.

14. Batch Navigation

Screen: FL-80

Workflow

1
Map shows waypoint route (multi-stop)
2
Current stop highlighted (1/2 badge)
3
Arrives at Stop 1 → taps "Arrived" → Collect + POD for this stop
4
Auto-advances to Stop 2
5
After last stop → back to Ready state

Design Notes

  • Stop counter badge on map (1/2, 2/2)
  • Current stop card below map with customer details
  • "Next" preview: rider knows what's coming
  • Each stop has independent COD collection + POD
  • If one stop fails (customer unavailable), other stops continue

15. COD Collection

Screen: FL-81

Purpose

Dedicated screen for cash collection. Big amount, quick-select buttons, running balance shown.

Design Notes

  • Amount to collect: HUGE font, center screen. No ambiguity.
  • Quick-select: "Exact", common denominations (100, 200)
  • Cash-in-hand balance: before and after this collection
  • Confirm button with amount repeated: "Confirm ₹72 Collected"
If customer pays partial or wrong amount, rider enters actual amount received. System flags discrepancy for manager review.

16. COD Threshold Block

Screen: FL-81

Purpose

Hard block. Rider cannot accept new orders until they settle cash above the threshold.

Workflow

1
Rider completes delivery, cash goes above threshold
2
This screen appears — cannot dismiss
3
Rider taps "Settle Now" → Settlement screen
4
After settlement confirmed → back to Ready state

Design Notes

  • Red status bar — visual urgency
  • Warning icon, clear language: "Settle Cash First"
  • Shows exact amounts: in hand vs threshold
  • Single action: "Settle Now" — no escape
  • Also triggers on: login with unsettled COD, logout attempt with cash
This is a BLOCKING screen. No bottom nav, no back button. Rider must settle. This is the #1 cash management control.

17. COD Settlement

Screen: FL-81

Workflow

1
Rider sees total cash to settle
2
Selects method: GPay, Bank, or Cash to Manager
3
For GPay: deep links to GPay with store UPI + amount
4
Enters transaction ID after payment
5
Submits → pending manager approval
6
Balance resets to 0 → rider unblocked

Design Notes

  • 3 settlement options, radio-style selection
  • GPay is default/recommended (fastest verification)
  • "Cash to Manager" = in-person handover, needs manager QR confirmation
  • Transaction ID field: for GPay/bank, manager verifies later
  • Optimistic: balance resets immediately, manager approves async

18. Earnings — Today

Screen: FL-83

Purpose

Full transparency on how today's pay is calculated. Rider trusts the system when they can see every line item.

Design Notes

  • Big number at top: total net earnings
  • Deliveries + hours worked as context
  • Breakdown: each component of the formula, line by line
  • Penalties in red with descriptions
  • Batch bonus highlighted in green
  • Cash status section: collected vs settled vs in hand
  • Tab toggle: Today / Weekly
Every number must be explainable. If rider asks "why ₹213 for distance?", they should be able to see: 14.2 km × ₹15/km = ₹213.

19. Earnings — Weekly

Screen: FL-83

Design Notes

  • Bar chart: daily earnings, 7 days. Today highlighted in yellow.
  • Total + deliveries + hours for the week
  • Settlement history: when, how much, method, status
  • "Unsettled" amount shown if any cash pending
  • Future days shown as empty bars (Sunday = day off)

20. Rate Card

Screen: FL-84

Purpose

Rider understands exactly how their pay works. Builds trust. Reduces disputes.

Design Notes

  • Formula shown in monospace, like an equation
  • Current rates in a clean table
  • Batch bonus rules explained with concrete examples
  • Example calculation: real numbers, step by step
  • "Full-time rates" badge — rider knows which tier they're on
  • Read-only — rider can't change anything here

21. Profile

Screen: FL-85

Design Notes

  • Photo + name + ID + type + status at top
  • Menu-style navigation to sub-sections
  • Referral link with bonus amount (growth lever)
  • "Check Out & End Shift" is red/outline — serious action, bottom of screen
  • Edit profile: opens editable version of profile setup

22. Check-out / End Shift

Screen: FL-82

Workflow

1
Rider taps "Check Out"
2
If COD balance > 0: blocked, redirected to settle
3
If settled: shift summary shown
4
Hours, deliveries, mandatory adherence, penalties, net pay
5
Confirm → checked out. App goes to login-ready state.

Design Notes

  • Shift summary: today's scorecard
  • Mandatory window adherence: green check or red X per block
  • COD status must be green (settled) before confirm is enabled
  • Red "Confirm Check-out" button — deliberate action
  • "Cancel" link to go back to shift
If rider hasn't completed 10h minimum, show warning: "You've only completed 9.8h. Leaving early may affect your record." But don't block — they can still check out.
