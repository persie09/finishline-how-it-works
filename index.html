Happy Path — everything works

This is the 90% case. An order comes in, a rider is found, the delivery completes, the customer is happy. No human involvement at any point. This is what "autonomous" means.

1
The trigger

A customer places an order

T+0 seconds

A customer in Andheri West opens BazaarNow's app and orders 3 items — milk, bread, eggs. The BazaarNow OMS processes the order and makes an API call to FinishLine.

What the OMS sends

POST /v1/orders with: hub_id (Andheri dark store), customer name, phone, address ("Shop 12, Andheri West Market"), delivery coordinates (lat/lng), items count (3), priority (standard).

What the system does immediately

1. Creates an order record in the database with status = created
2. Calculates the SLA deadline: now + 20 minutes (configurable per hub)
3. Emits an order.created event into the Event Bus
4. This single event triggers a cascade of 5 things simultaneously...

Event Bus Subscribers (in priority order)

Priority 0 — Event Logger: Writes the event to the immutable events table. This always runs first. The event is saved before anything else happens.

Priority 1 — State Updater: Updates orders.status = "created" in the orders table. This is the fast-read cache.

Priority 3 — Socket Broadcaster: Pushes a real-time update to the dashboard via WebSocket. If a hub manager is watching, they see the new order appear instantly.

Priority 5 — Webhook Dispatcher: Queues an outbound webhook if configured (for external integrations).

Priority 5 — Auto-Dispatch Handler: This is the big one. This triggers the Decision Engine. See next scene.

What the dashboard shows

The moment the order is created, the hub manager's dashboard updates:

Active orders counter: 4 → 5
The Action Feed shows a new entry
If the dashboard is in "Calm" mode, it stays calm — nothing is wrong yet.

All of this happens within ~50ms of the OMS API call.

Order #4821 created. SLA deadline: 14:20:00. System is now looking for a rider.
2
The brain kicks in

The Decision Engine finds a rider

T+0.1 seconds (100 milliseconds after order creation)

This is the core of the system. The auto-dispatch handler fires and the Decision Engine takes over. Here's exactly what happens inside:

Step 1 — Acquire the lock

The system grabs a Redis mutex lock on this specific order ID. Why? Because the SLA monitor, the escalation handler, and a human on the dashboard could all try to act on this order at the same time. The lock ensures only one decision-maker acts at a time. One brain, one order, one decision.

Step 2 — Find nearby riders

The system queries Redis GEO: "Give me all riders within 3km of the Andheri hub." (3km is the default radius — configurable per hub via the Config Engine.)

Redis GEO stores every online rider's last-known latitude/longitude. It returns a list with distances. Today, this returns 4 riders who are online and within range.

Step 3 — Filter for eligibility

Not every nearby rider can take this order. The system applies three filters:

Filter 1: Is the rider's status available? (Not on_trip, not on_break)
Filter 2: Is the rider below their max concurrent order limit? (Default: 1 order at a time)
Filter 3: Are they actually within the max radius? (Double-check against config)

Out of 4 nearby riders, 3 pass all filters. One is already on a trip — filtered out.

Step 4 — Score each eligible rider

This is the equity-aware dispatch algorithm. Every eligible rider gets a score from 0 to 1. The score is a weighted sum of three factors:

The scoring formula

Proximity score (weight: 0.4)
How close is the rider to the hub? Formula: 1 - (distance / max_radius)
A rider 0.5km away scores 0.83. A rider 2.5km away scores 0.17. Closer = higher.

Load balance score (weight: 0.3)
How busy is the rider right now? Formula: 1 - (active_orders / max_concurrent)
An idle rider scores 1.0. A rider at capacity scores 0.0. This prevents overloading.

Earnings equity score (weight: 0.3)
Has this rider gotten their fair share of work today? This is what makes the system different.
The system checks: how many deliveries has this rider done today vs. the hub average? How much has this rider earned today vs. the hub average?
equity = 0.5 x delivery_equity + 0.5 x earnings_equity
A rider who has done fewer deliveries than average scores higher. A rider who has earned less than average scores higher.
Underserved riders get priority. The research says: sacrificing 2.5% in delivery speed improves pay equity by 39%.

RiderDistanceProximity (0.4)Load (0.3)Equity (0.3)Final Score
Priya Sharma 0.8 km 0.73 1.00 0.85 0.85
Amit Kumar 0.5 km 0.83 1.00 0.40 0.75
Deepak Yadav 1.8 km 0.40 1.00 0.70 0.67
Why Priya wins, not Amit

Amit is closer (0.5km vs 0.8km), so he has a higher proximity score. But Amit has already completed 8 deliveries today — above the hub average of 6. His equity score is low (0.40).

Priya has only completed 3 deliveries today — well below average. Her equity score is high (0.85). Even though she's slightly further away, the equity weight (0.3) lifts her total score above Amit's.

This is the "2.5% cost sacrifice = 39% equity improvement" in action. We send the rider 300 metres further, but we balance earnings across the fleet. In a 12-person Tier 2 hub, this is the difference between a rider who stays and one who quits.

Step 5 — Execute the assignment

The Decision Engine has chosen Priya. Now it executes:

1. Decision logged: A full record is written to the decisions table — who was considered, scores, who won, why, the runner-up alternatives (Amit ranked #2, Deepak #3), estimated cost (₹18.40), estimated time (8 minutes), and a snapshot of the config that was active at this moment.

2. Order updated: status → assigned, assigned_rider_id → Priya's ID

3. Rider updated: Priya's status → on_trip, active_orders → 1

4. Events emitted: order.assigned + rider.assigned. These flow through the Event Bus again — logged, state updated, WebSocket pushed, webhook queued.

14:00:00.000 order.created → Order #4821, hub: Andheri, customer: Rahul Mehta
14:00:00.020 order.dispatching → Lock acquired, searching riders within 3km
14:00:00.089 decision.made → type: dispatch, action: assign_rider, rider: Priya Sharma, confidence: 0.85
14:00:00.102 order.assigned → rider: Priya Sharma, est. time: 8 min, est. cost: ₹18.40
14:00:00.102 rider.assigned → Priya Sharma, order #4821, distance to hub: 0.8km
Rider assigned in 102ms. Priya Sharma selected. No human involved. Decision fully auditable.

The pickup

Priya arrives at the hub and picks up the order

T+4 minutes

Priya rides to the Andheri dark store. She arrives, walks in, and the store staff hands her the bag with 3 items.

What triggers this in the system

Today (without a rider app): the hub manager or the API caller hits POST /v1/orders/:id/pickup. No request body needed.

In the future (with the rider app): Priya taps "Picked up" on her phone, which makes this same API call automatically.

What the system does

1. Validates state transition: order must be in assigned state to move to picked_up. If someone tries to pick up an order that's already in transit, the system rejects it.
2. Emits order.picked_up event → logged, state updated, dashboard pushed.
3. The SLA clock is still ticking. Priya has 16 minutes of the 20-minute window left.

Order picked up at 14:04. 16 minutes remaining on SLA.

The audit trail

Everything is permanently recorded

After delivery

This is the "sacred system of record" principle. Nothing is lost. Everything is queryable. If anyone asks "what happened with order #4821?" — the answer is complete.

What was recorded

6 events in the immutable events table — order.created, order.dispatching, order.assigned, order.picked_up, order.in_transit, order.delivered. Each with full payload, timestamps, actor, cost data.

1 decision in the decisions table — which riders were considered, their scores, who won, why, the alternatives, the config that was active, the cost estimate, the equity impact.

You can answer any question: "Why was Priya assigned and not Amit?" "How long did each step take?" "What was the cost?" "Was the SLA met?" "What config was active?" — all from the data, no guessing.

Happy path summary

Total time: 12 minutes 18 seconds. Human involvement: zero. Decision made in 102ms. 6 immutable events recorded. Full decision audit trail with runner-up alternatives. SLA met. Rider earnings tracked for equity. Dashboard updated in real-time throughout. Cost calculated and logged.

Exception Flow — No rider available

An order comes in, the Decision Engine searches for riders, and finds nobody. This is the most common exception in a thin Tier 2 fleet. Here's how the system self-heals.

1
Same start

Order created, dispatch triggered

T+0 seconds

Same as the happy path. Customer places order → OMS calls the API → order.created event → auto-dispatch handler fires.

The Decision Engine searches...

Redis GEO query: "riders within 3km of Andheri hub"
Result: 2 riders found

Eligibility filters:
- Rider A: status = on_trip → filtered out
- Rider B: status = on_break → filtered out

Zero eligible riders. Nobody can take this order right now.

No eligible riders within 3km. Decision Engine returns action: "no_rider".
2
Escalation triggered

The system escalates — but doesn't give up

T+0.2 seconds
What happens immediately

The auto-dispatch handler sees the "no_rider" response. It:

1. Sets order status → escalated
2. Emits order.escalated event
3. The Self-Heal Handler is listening for this event. It fires immediately.

Order escalated. Self-healing begins.
3
Self-healing step 1

Expand the search radius

T+0.3 seconds
What the Self-Heal Handler does

It takes the original radius (3km) and multiplies by 1.5x.
New search: "riders within 4.5km of Andheri hub"

This wider net catches riders who were just outside the original search.
The system re-runs the full dispatch logic — GEO query, eligibility filters, scoring — with the expanded radius.

Why 1.5x and not 2x or 3x?

If you expand too aggressively, you assign a rider 8km away — they take 15 minutes just to reach the hub, and the SLA is already half gone. 1.5x is a conservative first step. If this also fails, the escalation chain has more steps. Each step is configured in the Config Engine — you can change the multiplier per hub.

Scenario A — A rider is found at 4.2km:

Success — self-heal worked

The expanded search found Deepak Yadav at 4.2km. He's eligible. The system scores him (he's the only candidate, so he wins automatically), assigns him, and emits:

auto_heal.executed — "Self-healing succeeded. Rider found at expanded radius."
order.assigned — Deepak Yadav assigned

The order is back on track. The dashboard shows a brief amber flash (escalated, then resolved) and returns to green.

Self-heal succeeded. Deepak Yadav assigned at 4.2km. Order back on track.

Scenario B — Still no rider found:

Failure — expanded search also found nobody

Even at 4.5km, no eligible riders. The system emits:

auto_heal.exhausted — "All automatic recovery options exhausted."

The order stays in escalated status. This is now a human problem.

Self-heal exhausted. Human intervention required.
4
Human takes over

The hub manager sees the exception

T+1–2 minutes
What the hub manager sees

The Action Feed on the dashboard shows a red exception card:

"Order #4821 — No rider available. Auto-dispatch failed. Expanded radius failed. Needs manual assignment."

The manager now has to make a call. Their options:

Option: Manual assign

The manager sees the rider list and knows that Rider A is almost done with their current delivery (2 minutes away). They wait, then manually assign via POST /v1/orders/:id/assign. The system respects this — manual assignment overrides the engine.

Option: Cancel the order

If there's genuinely no way to deliver (all riders are stuck, weather is bad), the manager cancels via POST /v1/orders/:id/cancel. The order becomes terminal. The OMS is notified via webhook.

What the escalation chain is designed to be (configurable)

The default chain is configured in the Config Engine:

Step 1: Wait 120 seconds (maybe a rider will finish their current trip)
Step 2: Expand radius to 5km (configurable)
Step 3: Ping offline riders (not built yet — Phase 2)
Step 4: Alert human — surface to the dashboard exception queue

Today, only expand_radius and alert_human are implemented. Wait and ping_offline_riders are in the config schema but the execution logic isn't wired yet.

No-rider exception summary

The system tried automatically: expanded the search 1.5x. If that works, the order self-heals — no human needed. If it doesn't, the order escalates to the hub manager's exception queue. The full chain (wait → expand → ping offline → alert) is designed but only partially implemented in Phase 1.

Key insight for BazaarNow: In a 12-rider Tier 2 hub, this exception will happen more often than in a 200-rider Mumbai hub. The self-healing chain and its configuration per hub is survival-critical, not a nice-to-have.

Exception Flow — SLA breach

The order was assigned, the rider picked up, but things are taking too long. The 20-minute SLA window is closing. The system doesn't wait for it to breach — it warns proactively.

1
Normal start

Order created and assigned — all normal

T+0

Happy path until pickup. Rider assigned (Amit Kumar this time), order picked up at T+5 min. Rider heads out for delivery. SLA deadline is T+20 minutes.

Order in transit. 15 minutes remaining on SLA.
2
The SLA monitor wakes up

Warning: 70% of SLA consumed

T+14 minutes
What the SLA Monitor does (every 30 seconds)

The SLA Monitor is a BullMQ repeatable job that runs every 30 seconds. Each time, it:

1. Queries all active (non-terminal) orders from the database
2. For each order, calculates: elapsed_pct = (now - created_at) / (sla_target_at - created_at) × 100
3. Compares against thresholds

Order #4821 at T+14 minutes

elapsed_pct = 14/20 × 100 = 70%
Warning threshold (from config): 70%

The system emits: sla.warning
A Redis flag is set so this warning fires only once per order (no repeat noise).

The Decision Engine evaluates: should we escalate preemptively? (Config: preemptive_escalation: true)
Decision type: sla_action. If the order is still in transit, it flags for monitoring. If the order is still stuck at "assigned" (rider hasn't picked up), it may trigger escalation.

SLA warning at 70%. Order flagged. Dashboard shows amber status for this order. Hub manager is aware.
3
SLA breach

20 minutes have passed. Delivery not complete.

T+20 minutes
SLA Monitor detects breach

elapsed_pct = 100%+
The system emits: sla.breached
Again, Redis deduplication — fires only once.

Config says: breach_action: "escalate"
The order is marked as SLA-breached. The Decision Engine logs a decision of type sla_action with action escalate.

What the hub manager sees

The order card in the Action Feed turns red. "Order #4821 — SLA BREACHED. In transit for 21 minutes. Target was 20 minutes."

At this point the order is still being delivered — the rider is still riding. The breach is a metric event, not a cancellation. The delivery will still complete. But it counts against the on-time rate.

What is NOT built yet for SLA breach

Proactive customer notification — "Your order is running late, new ETA is X." This is in the architecture vision but not implemented. No SMS/push integration exists yet.

Automatic compensation — crediting the customer or adjusting the rider's metrics. Not implemented.

SLA recovery actions — like route optimisation or reassignment to a closer rider already heading that direction. Not implemented.

SLA breach summary

The SLA Monitor catches orders heading toward breach at 70% (warning) and 100% (breach). It logs everything. It flags the dashboard. It fires a decision. But the response is currently limited to flagging — there's no proactive customer communication, no route rerouting, no compensation engine. The monitoring is built. The remediation is not.

Exception Flow — Delivery failure

The rider reaches the customer, but can't deliver. Customer not home, wrong address, refused delivery. This is where the "fail" endpoint is used.

1
Normal flow until arrival

Rider reaches the address

T+10 minutes

Happy path up to this point. Order assigned, picked up, in transit. Rider arrives at the customer location. But the customer isn't there. Phone goes unanswered.

Rider at location. Customer not responding.
2
Failure recorded

Rider marks the delivery as failed

T+15 minutes
What triggers this

POST /v1/orders/:id/fail with a reason: "customer_not_available"

The system accepts this from assigned, picked_up, or in_transit states.

What the system does

1. Order status → failed (terminal state)
2. Rider freed: active_orders → 0, status → available
3. Event emitted: order.failed with failure_reason
4. Dashboard counter: failed orders +1, on-time rate unchanged (it wasn't delivered late — it wasn't delivered at all)

What is NOT built for failure recovery

Automatic re-dispatch: If the rider fails to deliver, the system does NOT automatically try to redeliver. The order goes terminal. A human would need to create a new order. This is a gap.

Customer contact before failure: The handover risk scoring exists (decision type: handover_risk) and would recommend "pre_contact_customer" for high-risk deliveries. But this recommendation is not wired to any notification system. The system scores the risk but can't act on it yet.

Return-to-hub flow: The rider has the items but the customer isn't there. There's no "return to hub" state or process in the current system. The rider just becomes available again. The items are... in limbo.

Delivery failure summary

The system can record a failure and free the rider. But it can't recover from it automatically — no re-dispatch, no return-to-hub flow, no customer notification. The handover risk scoring exists but can't act. This is the biggest gap between "autonomous operations" and reality. In Tier 2 with thin margins, every failed delivery costs ₹1,200+. Building the recovery loop is high-value.

Exception Flow — Order cancellation

The order needs to be cancelled. Could be: item out of stock, customer changed their mind, address is invalid. Cancellation can happen at different stages with different consequences.

1
Before assignment

Cancel while the system is still looking for a rider

Status: created or dispatching
What happens

POST /v1/orders/:id/cancel

Simplest case. No rider was assigned yet. The system:
1. Sets status → cancelled (terminal)
2. Emits order.cancelled event
3. If dispatch was in progress, the Redis lock ensures the cancel and the dispatch don't collide — one wins, one is rejected

Impact: Minimal. No rider was inconvenienced. No items were picked. Dashboard shows cancelled +1.

Cancelled cleanly. No rider involved. No wasted effort.
2
After assignment

Cancel after a rider was already assigned

Status: assigned
What happens

POST /v1/orders/:id/cancel

More complex. A rider (Priya) was already heading to the hub or is at the hub. The system:
1. Sets order status → cancelled
2. Frees the rider: Priya's active_orders → 0, status → available
3. Emits order.cancelled event

Impact: Priya wasted time riding to the hub. She's now available for the next order, but she did uncompensated work.

What is NOT built for post-assignment cancellation

Rider notification: Priya isn't automatically told the order was cancelled. Without a rider app, there's no channel to reach her. She'd find out when she arrives at the hub and the staff says "that order was cancelled."

Cancellation compensation: The system doesn't track or compensate for wasted rider time on cancelled orders. The equity score doesn't account for this — a cancelled order doesn't add to Priya's earnings but cost her time.

Cancellation reason tracking: The cancel endpoint doesn't require a reason (unlike the fail endpoint). Adding this would be valuable for identifying patterns — "40 cancellations this week due to out-of-stock" is an inventory signal.

Cancellation summary

Cancellation works mechanically — the order goes terminal, the rider is freed. But the rider communication gap means a rider could waste 10 minutes heading to a hub for a cancelled order. This is fixable once the rider app exists. For now, the hub manager would need to manually call the rider.

The mental model

Think of FinishLine today as a brain with no hands.

The brain is excellent. It makes the right dispatch decisions, tracks everything, self-heals when it can, and knows when to call for help. The scoring is real, the equity is real, the audit trail is real.

The hands are missing. No rider app means the brain can decide but can't communicate the decision to the rider directly. No customer notifications means the brain knows a delivery is late but can't tell the customer. No readiness signal means the brain dispatches immediately instead of waiting for the order to be packed.

Phase 2 builds the hands. Rider app, customer notifications, store readiness signals, failure recovery flows. The architecture is designed for this — every event, every decision, every config is already structured to support it. The brain doesn't need to be rebuilt. It just needs a body.
